## Context

Character standee images generated from the Character Sheet sometimes include model-produced sticker artifacts (white rectangular borders, frame strokes, and label/nameplate text). This conflicts with DungeonPlanner's intended rendering pipeline, where the white contour is generated by app postprocessing and should wrap only the character silhouette.

The current change already defines requirements for:
- Prompt constraints that explicitly forbid model-made border/frame/nameplate/text output.
- Postprocess cleanup that removes frame-like artifacts while preserving core character detail.
- No disruption to selection, placement, rendering, or serialization workflows.

Constraints:
- Keep implementation localized to character generation prompt composition, image postprocessing, and Character Sheet explanatory copy.
- Preserve existing standee contour generation behavior (postprocess-owned white outline).
- Avoid schema or save-format migrations.

## Goals / Non-Goals

**Goals:**
- Ensure generated prompts explicitly disallow borders, frames, nameplates, and text overlays.
- Improve subject isolation in postprocessing so frame-like connected components are removed before final standee output.
- Preserve the existing white contour result around the character silhouette.
- Keep placement and runtime behavior unchanged for processed characters.
- Add tests that protect against regressions in prompt wording and mask cleanup behavior.

**Non-Goals:**
- Redesigning standee geometry/material behavior in 3D scene rendering.
- Changes to model runtime/hosting configuration.
- Retrofitting already-generated character assets.

## Decisions

1. **Prompt recipe will carry explicit negative constraints for frame/text artifacts.**  
   - Rationale: The fastest reliability gain is reducing unwanted outputs at generation time.
   - Alternatives considered:
     - Only postprocess cleanup: rejected because it leaves avoidable noise in source imagery.
     - Generic "clean output" wording: rejected as too weak and model-dependent.

2. **Postprocess subject extraction will keep a dominant character component and suppress frame-like structures.**  
   - Approach: retain the primary subject component while filtering edge-connected thin rectangular structures and plate-like artifacts.
   - Rationale: Even with stronger prompts, some models can still emit border residue; cleanup must be robust.
   - Alternatives considered:
     - Tight global thresholding/cropping: rejected due to risk of clipping valid thin character features.
     - Manual user cleanup step: rejected as added UX friction.

3. **White outline remains exclusively postprocess-generated.**  
   - Rationale: This maintains visual consistency across assets and keeps contour styling under app control.
   - Alternatives considered:
     - Requesting model-generated sticker borders: rejected due to inconsistent style and frequent frame/nameplate artifacts.

4. **Character Sheet recipe copy will explicitly describe postprocess-owned borders.**  
   - Rationale: Aligns user-facing guidance with implementation to reduce confusion and prompt misuse.

5. **Regression tests will cover both prompt composition and frame-removal behavior.**  
   - Rationale: The change spans generation + postprocess; both surfaces need protection against future edits.

## Risks / Trade-offs

- **Over-aggressive artifact filtering may remove thin character details** → tune heuristics to favor silhouette preservation, and assert preservation-oriented scenarios in tests.
- **Under-aggressive filtering may leave residual border fragments** → include explicit frame/nameplate fixture cases in postprocess tests and iterate thresholds conservatively.
- **Prompt behavior can vary by model/version** → rely on dual-layer control (prompt constraints + deterministic postprocess cleanup), not prompt text alone.

## Migration Plan

1. Update prompt recipe composition and Character Sheet recipe copy in the app.
2. Update postprocess subject-isolation heuristics for frame/nameplate suppression.
3. Add/adjust tests for prompt constraints and postprocess frame-removal cases.
4. Run project test/build gates.
5. Rollback strategy (if needed): revert prompt/copy and postprocess heuristic changes together to restore previous generation-processing behavior.

## Open Questions

- Should frame artifact heuristics use fixed thresholds or image-size-normalized thresholds by default?
- Do we need additional synthetic fixtures for non-white frame artifacts in this change, or defer to a follow-up once white frame cases are stable?
